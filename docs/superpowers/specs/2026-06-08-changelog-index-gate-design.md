# Changelog Index Gate Design

Date: 2026-06-08

## Purpose

Prevent the update-log dialog from opening for versions that do not have a
real changelog section on the published docs site, while still allowing the
project to intentionally explain rollback or withdrawn-store versions when the
docs site publishes an entry for them.

The current update-log flow stores the current packaged extension version after
`runtime.onInstalled` reports an update. The first extension UI open consumes
that pending version and opens the inline changelog dialog for the matching
docs anchor. That flow does not verify whether the published docs site actually
contains a changelog section for the version.

## Current Context

The current extension version is read at runtime from the packaged manifest via
`browser.runtime.getManifest().version`. WXT derives the manifest version from
`package.json` during build unless overridden.

The update-log auto-open path is:

- background `runtime.onInstalled` receives an update event.
- the background reads the current manifest version.
- the background stores that version as `changelogOnUpdate_pendingVersion`.
- the first popup, options page, or sidepanel UI open consumes the pending
  version.
- `UpdateLogDialog` opens `changelog.html#<version-anchor>`.

This means an update, rollback, or store-side republish can create a pending
version even when the published docs page has no matching section. The iframe
can still load the changelog page successfully even when the hash target is
missing, so iframe load state is not a reliable section-existence check.

## Goals

- Treat the first available changelog version source as the source of truth,
  with the published docs-site index taking priority over GitHub fallbacks.
- Allow auto-opening changelog for a rollback or withdrawn version when the
  docs site intentionally includes that version in the index.
- Avoid auto-opening for rollback or same-version events when all changelog
  version sources cannot be fetched and no authoritative docs evidence is
  available.
- Preserve the existing update-log behavior for normal upgrades when all
  changelog version sources are temporarily unavailable.
- Keep manual changelog entry points available even when auto-open is skipped.
- Keep the docs index data-only, static, and generated by docs build tooling.
- Make the decision path testable without depending on a live docs deployment.

## Non-Goals

- Do not build a general product announcement or incident-notice system here.
- Do not rely on Chrome Web Store metadata to decide whether a changelog exists.
- Do not parse the rendered iframe DOM to detect a missing anchor.
- Do not block extension startup or UI opening on changelog-index networking.
- Do not make the update-log dialog show arbitrary remote content beyond the
  existing docs page.

## Changelog Version Sources

The docs build should generate a static JSON index from the Chinese changelog
source of truth:

```text
docs/docs/changelog.md
```

The published file should be available from the docs site as:

```text
/data/changelog-index.json
```

The first schema should stay small:

```json
{
  "schemaVersion": 1,
  "versions": ["3.44.0", "3.43.0", "3.42.0"]
}
```

Generation should extract version headings from the same source markdown that
VuePress renders for the public changelog page. A version is eligible for the
index only when the heading represents a real user-facing changelog section.
The extractor should normalize a leading `v` prefix, trim whitespace, and keep
only semver-like dotted numeric versions.

The index represents the currently published docs site's knowledge, not merely
the extension bundle's repository state. That is why it belongs to the docs
build output rather than only to extension build artifacts.

The extension should attempt changelog version sources in this order:

1. Published docs-site index:
   `https://all-api-hub.qixing1217.top/data/changelog-index.json`
2. GitHub raw index from `main`:
   `https://raw.githubusercontent.com/qixing-jk/all-api-hub/main/docs/docs/.vuepress/public/data/changelog-index.json`
3. GitHub raw changelog markdown from `main`:
   `https://raw.githubusercontent.com/qixing-jk/all-api-hub/main/docs/docs/changelog.md`

The first successfully fetched and parsed source is authoritative for the
decision. If the docs-site index is available and does not contain the current
version, the extension should not continue to GitHub raw sources looking for a
more favorable answer. This preserves the docs site's priority when it is
reachable, while still allowing GitHub raw fallbacks when the docs site or CDN
is unavailable.

GitHub Releases should not participate in this gate. A GitHub release can
exist without a matching docs changelog section, so it is suitable for
"latest version" checks but not for deciding whether to auto-open the docs
changelog dialog.

## Decision Rules

The extension should decide whether to record a pending update-log version with
this priority order:

1. Fetch and parse the published docs-site changelog index.
2. If it is available and valid, auto-open only when it contains the current
   extension version.
3. If the docs-site index is unavailable or invalid, fetch and parse the
   GitHub raw index.
4. If the raw index is available and valid, auto-open only when it contains the
   current extension version.
5. If the raw index is unavailable or invalid, fetch and parse the GitHub raw
   changelog markdown.
6. If the raw changelog markdown is available and valid, auto-open only when it
   contains the current extension version.
7. If all changelog version sources are unavailable or invalid, fall back to
   version-direction checks.
8. In fallback mode, skip auto-open only when `previousVersion` and
   `currentVersion` are both comparable and `previousVersion >= currentVersion`.
9. In fallback mode, allow auto-open for all other update events.

Pseudocode:

```ts
const source = await fetchFirstAvailableChangelogVersionSource()

if (source.ok) {
  return source.versions.includes(currentVersion)
}

const comparison = compareVersions(previousVersion, currentVersion)
if (comparison !== null && comparison >= 0) {
  return false
}

return true
```

The important product rule is that source membership overrides version
direction. A rollback version can still auto-open when the first available
source includes that version, because that means the project intentionally
published changelog or rollback guidance for it through an available
changelog source.

## Failure Behavior

The extension should fail open only for ordinary upgrade-like events where all
changelog version sources are unavailable. It should fail closed for explicit
rollback or same-version events without source evidence.

Examples:

| Situation | First available source result | Version comparison | Auto-open |
| --- | --- | --- | --- |
| Normal release with docs entry | contains current | any | yes |
| Normal release missing from reachable docs index | valid docs index but missing | any | no |
| Docs index unavailable, raw index contains current | contains current | any | yes |
| Docs and raw index unavailable, raw markdown contains current | contains current | any | yes |
| Rollback with docs explanation | contains current | previous >= current | yes |
| Rollback with no explanation in reachable source | valid source but missing | previous >= current | no |
| Rollback while all sources unavailable | unavailable | previous >= current | no |
| Normal upgrade while all sources unavailable | unavailable | previous < current | yes |
| Missing `previousVersion` while all sources unavailable | unavailable | unknown | yes |
| Unparseable version while all sources unavailable | unavailable | unknown | yes |

Invalid index JSON, unsupported schema, non-array `versions`, or invalid version
entries should be treated as source unavailable and logged at warning level.
Invalid raw changelog markdown should also be treated as source unavailable.
These failures should not create a hard runtime failure.

## Integration Points

Docs side:

- Add a docs-build script that extracts versions from `docs/docs/changelog.md`.
- Write `docs/docs/.vuepress/public/data/changelog-index.json`.
- Add a docs check that verifies the generated index is up to date before docs
  deployment.

Extension side:

- Add a small changelog-index client under the updates domain.
- Use a bounded fetch timeout so update handling is not held indefinitely.
- Add helpers for the published docs index URL, GitHub raw index URL, and
  GitHub raw changelog markdown URL.
- Try changelog sources in priority order and stop at the first successfully
  parsed source.
- Apply the decision before `changelogOnUpdateState.setPendingVersion`.
- Keep `ChangelogOnUpdateStateService` as a simple pending-version store.

## Security & Privacy

The changelog-index client fetches only fixed project-owned documentation
sources: the published docs index, the GitHub raw index, and the GitHub raw
changelog markdown exposed by the docs-link helpers. Browser CSP/CORS must allow
those fixed origins; a blocked or failed response is treated as an unavailable
source rather than a runtime failure.

The bounded 2,000ms fetch timeout limits update-flow latency and avoids
amplifying release-day request spikes. The first implementation does not add
retry/backoff/queueing; each update checks each source at most once before
`ChangelogOnUpdateStateService` and `changelogOnUpdateState.setPendingVersion`
use the fallback decision. Requests do not include account URLs, API keys,
tokens, prompts, or user-entered text; only ordinary browser request metadata is
sent to the fixed docs/GitHub raw origins.

## Testing

Unit tests should cover:

- markdown heading extraction for version headings.
- generated index shape and version normalization.
- index and markdown parser success and failure cases.
- source ordering, including "docs source missing current version stops before
  raw fallback".
- auto-open decision table, including rollback-with-source and
  rollback-without-source cases.
- background `onInstalled` behavior with `previousVersion`.

Component tests for `UpdateLogDialog` do not need to change for the first
implementation because the dialog still receives a version and renders the same
docs URL. End-to-end coverage is not required unless implementation introduces
new browser-runtime behavior beyond the existing update event and storage path.

## Open Questions Resolved

- The first available changelog version source is authoritative, even for
  rollback or same-version events.
- Source order is docs-site index, GitHub raw index, then GitHub raw changelog
  markdown.
- `previousVersion >= currentVersion` is not an absolute blocker. It is only a
  fallback rule when all changelog version sources cannot be fetched or parsed.
- Changelog source fetch failure should not suppress normal upgrade changelog
  prompts.
- Manual changelog links remain available regardless of auto-open decisions.
